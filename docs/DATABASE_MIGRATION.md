## Fix for postWords Column Truncation

### Issue
The `postWords` column in the `posts` table is too small to store long JSON arrays of word tokens, causing data truncation errors when inserting posts with lengthy content.

#### Error Message
```
执行 SQL 失败: ('42000', '[42000] [Microsoft][ODBC Driver 17 for SQL Server][SQL Server]字符串或二进制数据将在表"SSPU.dbo.posts"，列"postWords"中被截断。截断值:"["中国", "高等教育", "学生", "信息网", "（", "学信网", "）", "将", "于", "2025", "年", "10", "月", "15", "日至", "12", "月""。 (2628) (SQLParamData); [42000] [Microsoft][ODBC Driver 17 for SQL Server][SQL Server]语句已终止。 (3621)')
```

This occurs because:
1. The `postWords` column stores a JSON array of Chinese word tokens generated by jieba
2. For long articles, the JSON array can be very large (thousands of characters)
3. The column was defined without specifying a size, defaulting to VARCHAR(255) or similar
4. Long content gets truncated, causing the SQL insert to fail

### Solution

#### Option 1: Update Existing Column (Recommended)

Execute this SQL to alter the existing column to NVARCHAR(MAX):

```sql
-- Alter the postWords column to NVARCHAR(MAX)
ALTER TABLE dbo.posts 
ALTER COLUMN postWords NVARCHAR(MAX) NULL;
```

#### Option 2: Recreate Table (If Option 1 Fails)

If you encounter issues with the ALTER command, you may need to recreate the table:

```sql
-- Step 1: Backup existing data
SELECT * INTO posts_backup FROM posts;

-- Step 2: Drop the existing table
DROP TABLE posts;

-- Step 3: Create the table with correct column definitions
CREATE TABLE posts (
    id BIGINT IDENTITY(1,1) PRIMARY KEY,
    postName NVARCHAR(MAX),
    postReleaseTime NVARCHAR(255),
    postSource NVARCHAR(MAX),
    postContent NVARCHAR(MAX),
    postUrl NVARCHAR(MAX),
    postFiles NVARCHAR(MAX),
    postContentUsingMarkdown NVARCHAR(MAX),
    postSimplifiedContent NVARCHAR(MAX),
    postWords NVARCHAR(MAX)
);

-- Step 4: Restore data with IDENTITY_INSERT
SET IDENTITY_INSERT posts ON;
INSERT INTO posts (id, postName, postReleaseTime, postSource, postContent, postUrl, postFiles, postContentUsingMarkdown, postSimplifiedContent, postWords)
SELECT id, postName, postReleaseTime, postSource, postContent, postUrl, postFiles, postContentUsingMarkdown, postSimplifiedContent, postWords 
FROM posts_backup;
SET IDENTITY_INSERT posts OFF;

-- Step 5: Drop backup table
DROP TABLE posts_backup;
```

#### Option 3: Let Hibernate Update (Development Only)

If you're in development without important data:

1. **Backup any important data**
2. **Temporarily change application properties:**
   ```properties
   spring.jpa.hibernate.ddl-auto=create-drop
   ```
3. **Restart the application** - This will drop and recreate the table
4. **Change back to update:**
   ```properties
   spring.jpa.hibernate.ddl-auto=update
   ```

### Verification

After applying the migration, verify the column type:

```sql
-- Check the postWords column definition
SELECT 
    COLUMN_NAME,
    DATA_TYPE,
    CHARACTER_MAXIMUM_LENGTH,
    IS_NULLABLE
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'posts' AND COLUMN_NAME = 'postWords';
```

Expected result:
- DATA_TYPE: `nvarchar`
- CHARACTER_MAXIMUM_LENGTH: `-1` (indicates MAX)

## Fix for Admin Page - Bots Table IDENTITY

### Issue
The `bots` table in SQL Server needs to have its `id` column configured as an IDENTITY column to support auto-increment functionality required by the admin page.

### Error Message
```
org.springframework.dao.DataIntegrityViolationException: could not execute statement 
[不能将值 NULL 插入列 'id'，表 'SSPU.dbo.bots'；列不允许有 Null 值。INSERT 失败。]
```

This occurs because:
1. The Bot entity has `@GeneratedValue(strategy = GenerationType.IDENTITY)`
2. But the existing database table doesn't have IDENTITY configured
3. When inserting, Hibernate doesn't provide an ID (expects DB to generate it)
4. SQL Server rejects the insert because the column doesn't allow NULL and has no default/identity

## Solution Options

### Option 1: Quick Fix - Temporary Schema Recreation (Development Only)

If you're in development and don't have important data:

1. **Backup important data if any exists**
2. **Temporarily change application-prod.properties:**
   ```properties
   # Change from:
   spring.jpa.hibernate.ddl-auto=update
   
   # To:
   spring.jpa.hibernate.ddl-auto=create-drop
   ```
3. **Restart the application** - This will drop and recreate all tables with correct IDENTITY
4. **Change back to update:**
   ```properties
   spring.jpa.hibernate.ddl-auto=update
   ```
5. **Restart again** - Normal operation resumes

### Option 2: Manual SQL Fix (Recommended for Production)

#### For Existing Database with Data

```sql
-- Step 1: Backup existing data
SELECT * INTO bots_backup FROM bots;

-- Step 2: Drop the existing table
DROP TABLE bots;

-- Step 3: Create the table with IDENTITY column
CREATE TABLE bots (
    id BIGINT IDENTITY(1,1) PRIMARY KEY,
    name NVARCHAR(255) NOT NULL UNIQUE,
    description NVARCHAR(MAX),
    avatarUrl NVARCHAR(MAX),
    systemPrompt NVARCHAR(MAX) NOT NULL,
    selectedModel NVARCHAR(MAX),
    apiKey NVARCHAR(MAX),
    baseUrl NVARCHAR(MAX),
    isActive BIT NOT NULL,
    createdAt NVARCHAR(255) NOT NULL,
    updatedAt NVARCHAR(255) NOT NULL,
    temperature FLOAT,
    topK INT
);

-- Step 4: Restore data with IDENTITY_INSERT
SET IDENTITY_INSERT bots ON;
INSERT INTO bots (id, name, description, avatarUrl, systemPrompt, selectedModel, apiKey, baseUrl, isActive, createdAt, updatedAt, temperature, topK)
SELECT id, name, description, avatarUrl, systemPrompt, selectedModel, apiKey, baseUrl, isActive, createdAt, updatedAt, temperature, topK 
FROM bots_backup;
SET IDENTITY_INSERT bots OFF;

-- Step 5: Drop backup table
DROP TABLE bots_backup;
```

#### For New Database (No Existing Data)

```sql
-- Simply drop and let Hibernate recreate
DROP TABLE IF EXISTS bots;

-- Then restart the application, Hibernate will create it correctly
```

### Option 3: Add New Columns to Existing Bot Table

If you just added `temperature` and `topK` columns and the table already exists:

```sql
-- Add the new columns if they don't exist
IF NOT EXISTS (SELECT * FROM sys.columns WHERE object_id = OBJECT_ID(N'bots') AND name = 'temperature')
BEGIN
    ALTER TABLE bots ADD temperature FLOAT NULL;
END

IF NOT EXISTS (SELECT * FROM sys.columns WHERE object_id = OBJECT_ID(N'bots') AND name = 'topK')
BEGIN
    ALTER TABLE bots ADD topK INT NULL;
END

-- Fix the ID column to be IDENTITY (requires table recreation)
-- Follow Option 2 steps above
```

## Spider Table Schema

The `spiders` table also needs to be created with IDENTITY:

```sql
CREATE TABLE spiders (
    id BIGINT IDENTITY(1,1) PRIMARY KEY,
    name NVARCHAR(255) NOT NULL UNIQUE,
```

## Verification

After applying the migration, verify the IDENTITY is set:

```sql
-- Check if IDENTITY is configured
SELECT 
    COLUMN_NAME,
    IS_NULLABLE,
    DATA_TYPE,
    COLUMNPROPERTY(OBJECT_ID(TABLE_SCHEMA+'.'+TABLE_NAME), COLUMN_NAME, 'IsIdentity') as IsIdentity
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'bots' AND COLUMN_NAME = 'id';
```

The `IsIdentity` column should return `1`.

## Common Issues

### Issue: "Cannot insert NULL into column 'id'"
**Cause:** The `id` column is not configured as IDENTITY  
**Solution:** Use Option 1 or Option 2 above

### Issue: "Column 'temperature' does not exist"
**Cause:** The table exists but new columns weren't added  
**Solution:** Use Option 3 or let `spring.jpa.hibernate.ddl-auto=update` add them

### Issue: Data loss after recreation
**Cause:** Didn't backup data before dropping table  
**Solution:** Always backup first! Use the backup steps in Option 2

## Notes for Production

- **Never use `create-drop` in production** - it deletes all data on each restart
- **Use `update` mode** - safest for production, but may not fix schema issues
- **Use migration tools** - Consider Flyway or Liquibase for version-controlled migrations
- **Backup first** - Always backup before schema changes
- **Test in dev** - Test all migrations in development before applying to production

## Current Configuration

The project is configured with:
```properties
spring.jpa.hibernate.ddl-auto=update
```

This is safe and will:
- Create new tables if they don't exist
- Add new columns if they don't exist
- **NOT** modify existing columns (won't add IDENTITY to existing id column)
- **NOT** drop tables or data

For the bot creation to work, you must manually fix the IDENTITY issue using one of the options above.
